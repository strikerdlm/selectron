## Stage-A Disposition

Stage A is frozen as a methodological demonstration until a separate ratification
process is completed. The active catalog is:

- canonical module: `src/data/demo-criteria.ts`;
- status: `demo`;
- intended use: uncertain-weight MCDA demonstration only;
- non-use: no real select/reject recommendation, no validated suitability ranking,
  no medical clearance, and no NASA-equivalent verdict.

Compatibility imports from `src/data/placeholder-criteria.ts` are retained as a
shim for historical tests and documents, but new code should import from
`@/data/demo-criteria`.

To graduate Stage A from demo to ratified, a future release must add:

1. a versioned expert-review record for every criterion;
2. a documented weighting-elicitation process;
3. multi-candidate rank/tie acceptance tests;
4. a user-facing label that distinguishes MCDA assumptions from empirical outcome validation.

## Evidence And Validation Protocol

K15 is a verification benchmark only. It tests agreement with published IMM
reference outputs under specified assumptions. It is not validation against
observed analog-mission clinical outcomes.

External validation must use observed analog outcomes and be tracked separately:

| Step | Requirement |
|---|---|
| Cohort definition | Mission analog type, duration, crew size, medical-support tier, inclusion/exclusion rules. |
| Endpoint definition | TME, EVAC-equivalent, LOCL-equivalent, duty-time lost, and condition-specific incident definitions must be mapped before fitting. |
| Data provenance | Every included row must have source, denominator, person-time/exposure, extraction quote, extractor, verifier, risk of bias, and transportability note. |
| Holdout design | At least one mission/source family is held out from prior construction before predictive comparison. |
| Calibration metrics | Compare predicted intervals and observed rates with coverage, calibration slope/intercept where feasible, and predeclared divergence handling. |
| Claim boundary | Passing this protocol can support analog-outcome predictive claims only within the validated mission/source family. |

Adjudication priority is by decision influence, not by treating all 4,846 numeric
parameter paths equally. Initial priority order:

1. global calibration multipliers and mission-kind multipliers;
2. high-sensitivity incidence parameters from Sobol/Morris output;
3. outcome parameters driving pEVAC and pLOCL;
4. severity branches for persistent-impairment conditions;
5. resource/treatment parameters with high kit-scenario leverage;
6. low-sensitivity tail parameters.

## Architecture And CI Gate

The active app is `src/engine`, `src/imm`, `src/ui/views/{Dashboard,Wizard,CrewComposition,Calibration,Analysis}`,
`src/contexts`, `src/db`, `src/workers`, and `src/api`.

`src/risk` is archived. It may still be imported by:

- tests under `tests/risk`;
- historical/manuscript figure harnesses;
- compatibility wrappers that re-export neutral engine utilities.

It must not be reintroduced into active Wizard, Dashboard, Crew Composition,
Calibration, Analysis, IMM simulation, or worker paths. The CI command
`npm run guard:active-imports` enforces this boundary.

## Release And Publication Freeze

The active development version is **v0.6.0-rebaseline.0**. The earlier changelog
entry for 0.5.7 is labeled unreleased because source metadata never advanced to
0.5.7 as a version of record.

Before any methods/software paper package is regenerated or uploaded:

1. tag the source commit that passed CI;
2. run the required CI lanes and record URLs/statuses;
3. regenerate evidence status with `npm run evidence:status -- --write`;
4. rebuild manuscript/submission assets from the frozen tag only;
5. remove transient browser captures and generated files from ordinary source review;
6. perform a final claim audit against this document and README Known Limits.
