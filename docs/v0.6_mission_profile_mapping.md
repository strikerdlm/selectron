# v0.6 — Mission-profile → modeled-effect mapping

**Status:** Containment registry implemented; effect-estimate population remains
a research-design workstream.
**Registry version:** `2026-06-v0.6-rebaseline`
**Source of truth (current):** `src/imm/profile-effects.ts`

## The gap (review finding F2)

The `AnalogMissionProfile` (`src/imm/types.ts`) carries rich isolation and
confinement context — habitat privacy, sleeping arrangement, workload,
circadian lighting, autonomy, communication delay, hygiene constraint,
evacuation time, and more. None of these fields drive `simulateIMM`. The
simulator is currently driven by:

- `mission.durationDays` (Poisson exposure time),
- `mission.kind` (per-(kind, condition) incidence multipliers from
  `imm-priors.json::global_calibration.kind_multipliers`),
- `mission.evaSchedule` (space-EVA-coupled condition timing for ISS/future space runs; excluded for terrestrial analogs),
- `mission.crewSize` / `mission.totalEVAs` (exposure denominators),
- the medical `kit` (resource-availability factor + delivery-class gating).

Consequently the app estimates outcomes for **duration-plus-prior scenarios**;
it does not yet **quantify isolation or confinement exposure itself**. A
30-day high-workload high-autonomy polar profile and a 30-day low-workload
low-autonomy controlled profile produce identical IMM outputs if they share
duration, kind, space-EVA schedule handling, and kit. Terrestrial `profile.eva`
values such as `terrain-field` and `polar-field` are descriptive until
analog-specific exposure denominators and priors are added.

## Implemented containment mapping

An explicit, versioned `ProfileEffect` registry (`src/imm/profile-effects.ts`)
maps each profile field to a modeled effect:

```ts
type ProfileEffect = {
  profilePath: string;             // e.g. "profile.operations.workload"
  conditionId?: string;             // per-condition, else global
  target: "incidence" | "severity" | "treatment-delay" | "recovery";
  model: "log-hazard" | "odds-ratio" | "time-shift";
  estimate: number | null;           // null while unsupported
  uncertainty?: unknown;             // CI or distribution
  evidenceStatus: "accepted" | "proposal" | "unsupported";
  sourceIds: string[];               // research/ slugs or citation DOIs
  note: string;                       // claim-boundary text
};
```

Every field is registered today and exported in three explicit groups:
`ACCEPTED_PROFILE_EFFECTS`, `PROPOSAL_PROFILE_EFFECTS`, and
`DESCRIPTIVE_ONLY_FIELDS`.

## Current effect split

| group | profilePath values | active in default/adjudicated runs? | claim boundary |
|---|---|---:|---|
| accepted mechanical effects | `mission.durationDays`; `mission.evaSchedule` | yes | Exposure-time and scheduled space-EVA mechanics only; not external analog validation |
| proposal-stage effects | `mission.kind`; `kit.resources`; `profile.communication.delaySec` | no, except when explicitly exploratory/custom | Sensitivity inputs only; unadjudicated, not release-prior evidence |
| descriptive-only fields | `profile.habitat.privacyLevel`; `profile.habitat.sleepingArrangement`; `profile.operations.workload`; `profile.operations.circadianLightControl`; `profile.operations.autonomyLevel`; `profile.logistics.hygieneConstraint`; `profile.medicalSupport.evacuationTimeHours`; `profile.eva.type` | no | Stored and displayed for context; no modeled effect |

Terrestrial `profile.eva.type` is explicitly descriptive-only because
terrain-field and polar-field hazards require separate analog-specific exposure
denominators and priors. Other isolation/confinement fields remain descriptive
until an evidence search, transportability note, and prior-adjudication path
are completed.

## Candidate effect models (to be populated with evidence)

| profilePath | target | candidate model | candidate evidence base |
|---|---|---|---|
| `profile.habitat.privacyLevel` | incidence | log-hazard | Antarctic / submarine crew behavioral-health literature |
| `profile.habitat.sleepingArrangement` | incidence | log-hazard | analog sleep-quality → fatigue-related incidence |
| `profile.operations.workload` | incidence | log-hazard | workload → injury / error incidence |
| `profile.operations.circadianLightControl` | incidence | log-hazard | circadian disruption → fatigue/performance incidence |
| `profile.operations.autonomyLevel` | treatment-delay | time-shift | autonomy → remote-guidance latency |
| `profile.communication.delaySec` | treatment-delay | time-shift | comms delay → treatment-delay time-shift (Mars DRMs) |
| `profile.logistics.hygieneConstraint` | incidence | log-hazard | hygiene → infection / skin incidence |
| `profile.medicalSupport.evacuationTimeHours` | severity | log-hazard | evacuation time → severe-outcome hazard |
| `profile.eva.type` | incidence | analog exposure family | terrain-field / polar-field / habitat-egress injury and operations denominators |

Each candidate requires an evidence search, an elicited estimate with
uncertainty, and an `evidenceStatus` upgrade (typically `proposal`, since
analog outcome registries are not yet available — see `STATUS.md` and the
`releasePriorsAdjudicated=false` evidence ledger).

## Decisions the author must make

1. **Scope and priority** of which profile fields to model first (privacy and
   workload have the strongest analog behavioral-health evidence; comms delay
   is the Mars-DRM prerequisite per `docs/future_features.md`).
2. **Effect model family** per field (log-hazard vs odds-ratio vs time-shift)
   and whether effects are global or per-condition (`conditionId`).
3. **Evidence source policy**: which literature is admissible as `proposal`
   vs requiring an adjudicated ledger row for `accepted`.
4. **UI presentation**: whether unsupported fields should render at all in a
   scientific run, or be hidden until modeled.
